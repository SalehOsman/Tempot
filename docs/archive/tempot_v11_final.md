# 1. نظرة عامة على Tempot

> **In a Nutshell:** Tempot إطار عمل جاهز لبناء بوتات تيليجرام احترافية. يوفر كل البنية التحتية — الصلاحيات، الجلسات، الذكاء الاصطناعي، الإشعارات، التوثيق — بحيث يركز المطور على منطق الأعمال فقط.

Tempot اسم مشتق من كلمتي Template وBot، وهو إطار عمل مؤسسي متكامل لبناء بوتات تيليجرام احترافية. يوفر القالب كل البنية التحتية جاهزة للاستخدام الفوري، بحيث يركز المطور على منطق الأعمال فقط دون إعادة بناء المكونات الأساسية في كل مشروع.

يعتمد Tempot على مبدأ المعمارية النظيفة مع فصل تام بين الطبقات، ونظام موديولات ديناميكي قابل للتوسع، وأدوات ذكاء اصطناعي مدمجة عبر طبقة تجريد موحدة، ومعايير UX واضحة لضمان تجربة مستخدم احترافية عبر كل الموديولات.

### 1.1 مبدأ "عدم إعادة اختراع العجلة"

> ⚠️ **مبدأ معماري أساسي:** كل مكون في Tempot يعتمد على مكتبات مختبرة ونشطة التطوير بدلاً من إعادة بنائها من الصفر. القاعدة: إذا وُجدت مكتبة مفتوحة المصدر بأكثر من 500 نجمة على GitHub، نشطة خلال 6 أشهر، ومتوافقة مع TypeScript وNode.js 22.12+ — تُستخدم بدلاً من البناء المخصص.

# 3. المعمارية العامة

> **In a Nutshell:** معمارية ثلاثية الطبقات (نواة ← خدمات ← واجهات) مع 5 مبادئ معمارية إلزامية تضمن عدم التكرار والقابلية للتوسع.

| الحزمة          | المسار                              | تعتمد على                                  | اختياري                 |
| --------------- | ----------------------------------- | ------------------------------------------ | ----------------------- |
| ai-core         | `/packages/ai-core`                 | cache-manager, Drizzle                     | نعم (`TEMPOT_AI=false`) |
| cache (wrapper) | `/packages/shared/cache.ts`         | cache-manager, @keyv/redis, @keyv/postgres | لا — أساسي              |
| queue (factory) | `/packages/shared/queue.factory.ts` | BullMQ, Redis                              | لا — أساسي              |

### 3.1 الطبقات الثلاث

| الطبقة        | المحتوى                                   | المسؤولية                            |
| ------------- | ----------------------------------------- | ------------------------------------ |
| طبقة النواة   | منطق الأعمال، الموديولات، الخدمات         | القرارات التجارية والعمليات الأساسية |
| طبقة الخدمات  | قواعد البيانات، التخزين، الذكاء الاصطناعي | التعامل مع الموارد الخارجية          |
| طبقة الواجهات | تيليجرام، الويب، لوحة التحكم              | التفاعل مع المستخدم النهائي          |

### 3.2 مبادئ معمارية إلزامية

> ⚠️ هذه المبادئ غير قابلة للتفاوض — كل كود يخالفها يُرفض في Code Review.

1. **طبقة تجريد لكل مورد خارجي** — كل خدمة خارجية (AI، Storage، Payment) تُغلَّف بـ interface يسمح بالتبديل.
2. 🆕 **Cache Layer موحدة عبر cache-manager** — مكتبة `cache-manager` مع Keyv adapters بدلاً من بناء حزمة cache مخصصة. Thin wrapper في `packages/shared/cache.ts` يوفر API موحد.
3. 🆕 **Queue Factory مركزي** — factory function في `packages/shared/queue.factory.ts` تُوحّد إعداد BullMQ (~30 سطر) بدلاً من حزمة مستقلة.
4. **Event-Driven بين الموديولات** — التواصل بين الموديولات حصراً عبر Event Bus.
5. 🆕 **Graceful Degradation** — كل خدمة خارجية لها سيناريو تدهور موثق ومُنفَّذ.

> 📄 **للرسوم البيانية المرئية:** راجع ملف `docs/architecture/ARCHITECTURE-DIAGRAMS.md` — يشمل 11 رسم Mermaid: C4 Levels 1-3, Layer Diagram, Event Flow, Module Anatomy, Security Chain, وأكثر.

# 5. نظام الصلاحيات — CASL-Based

> **In a Nutshell:** نظام صلاحيات رباعي المستويات (GUEST → USER → ADMIN → SUPER_ADMIN) مبني على CASL بدلاً من بناء مخصص. يدعم Scoped Authorization لتحديد أدمنز معينين لكل موديول.

| الحزمة    | المسار                | تعتمد على    | اختياري                   |
| --------- | --------------------- | ------------ | ------------------------- |
| auth-core | `/packages/auth-core` | CASL, Prisma | نعم (`TEMPOT_AUTH=false`) |

> 📐 **ADR-013:** اعتماد CASL بدلاً من بناء RBAC مخصص — البديل المرفوض: بناء نظام صلاحيات كامل من الصفر (تكلفة صيانة عالية، لا اختبارات مجتمعية). CASL: 27k+ ⭐، Prisma adapter جاهز، ~50KB.
> 🔄 **تغيير عن v10:** كان النظام مبنياً من الصفر بالكامل.

## 5.1 مستويات الأدوار

| الدور       | المستوى    | الوصف                                                 |
| ----------- | ---------- | ----------------------------------------------------- |
| SUPER_ADMIN | 4 — الأعلى | صلاحية مطلقة، يتجاوز جميع القيود تلقائياً دون استثناء |
| ADMIN       | 3          | يدخل الموديولات المخصصة له عبر Scoping أو بدونه       |
| USER        | 2          | يدخل الموديولات المفتوحة لمستوى المستخدم              |
| GUEST       | 1 — الأدنى | أدنى مستوى، يدخل موديولات محددة فقط                   |

## 5.2 تعريف الصلاحيات عبر CASL

🔄 كل موديول يُعرّف صلاحياته في ملف `abilities.ts` باستخدام CASL `defineAbility`:

```typescript
// مثال: تعريف صلاحيات موديول الفواتير
import { defineAbility } from '@casl/ability';
import { accessibleBy } from '@casl/prisma';

export const defineInvoiceAbilities = (user: SessionUser) => {
  return defineAbility((can, cannot) => {
    // SUPER_ADMIN — God Mode
    if (user.role === 'SUPER_ADMIN') {
      can('manage', 'all');
      return;
    }
    // ADMIN — حسب Scoping
    if (user.role === 'ADMIN') {
      can('read', 'Invoice');
      can('create', 'Invoice');
      can('update', 'Invoice', { createdBy: user.id });
    }
    // USER — قراءة فقط
    if (user.role === 'USER') {
      can('read', 'Invoice', { userId: user.id });
    }
  });
};
```

## 5.3 قواعد الوصول

- ⚠️ SUPER_ADMIN يملك صلاحية مطلقة دائماً ويتجاوز أي قيد بدون استثناء — God Mode عبر `can('manage', 'all')`.
- كل موديول يحدد مستوى الصلاحية المطلوب إلزامياً في ملف الإعدادات، والافتراضي ADMIN.
- يدعم النظام Scoped Authorization عبر CASL conditions: تحديد أدمن معين أو أكثر للموديول بدلاً من كل الأدمنز.
- إن لم يُعرَّف Scoping فكل من يملك الدور المطلوب يدخل الموديول.
- إن عُرِّف Scoping فقط المحددون بالاسم يدخلون حتى لو كانوا يملكون الدور.
- كل محاولة وصول مرفوضة تُسجَّل تلقائياً في Audit Log (القسم 10.3).
- المحاولات المتكررة المرفوضة من نفس المستخدم ترسل تنبيهاً فورياً للسوبر أدمن.
- الموديولات ذات `isCore: true` محمية على مستوى النظام ولا يمكن تعطيلها حتى بواسطة SUPER_ADMIN. هذا القيد تقني وليس صلاحياتياً — تعطيل هذه الموديولات يُوقف النظام كلياً.

## 5.4 التكامل مع Prisma عبر @casl/prisma

```typescript
// استعلام Prisma مع فلترة تلقائية حسب الصلاحيات
const invoices = await prisma.invoice.findMany({
  where: accessibleBy(ability, 'read').Invoice,
});
```

## 5.5 تعريف SUPER_ADMIN الأول

يتم تعريف Super Admin الأول عبر متغير البيئة `SUPER_ADMIN_IDS` في ملف `.env` حماية من حالة غياب أي سوبر أدمن عند أول تشغيل. يمكن تعريف أكثر من سوبر أدمن مفصولين بفاصلة.

# 6. نظام إدارة المستخدمين

| الحزمة               | المسار             | تعتمد على                  | اختياري |
| -------------------- | ------------------ | -------------------------- | ------- |

## 6.1 أوضاع الانضمام

يدعم النظام أربعة أوضاع للانضمام يُتحكم فيها من لوحة الإعدادات دون إعادة تشغيل:

| الوضع       | الوصف                               | من يوافق                   |
| ----------- | ----------------------------------- | -------------------------- |

- يمكن تحديد تاريخ انتهاء صلاحية الرابط (`null` = لا ينتهي).
- الرابط يُلغى تلقائياً عند انتهاء الاستخدام أو تاريخ الانتهاء.

يعتمد Tempot على استراتيجية مزدوجة: Redis للسرعة في القراءة المتكررة، وPostgreSQL للديمومة والاستعلام التاريخي. طبقة SessionManager المركزية تُخفي هذا التعقيد عن الموديولات تماماً.

> ⚠️ **Dynamic Input Engine حزمة Core مركزية مبنية فوق grammY Conversations وZod. تقضي على تكرار كتابة تدفقات المحادثة — المطور يعرّف Schema للحقول، والمحرك يتولى الحوار والتحقق وإعادة بيانات نظيفة Type-Safe.**

### 7.3.5 قواعد التوافق مع معايير Tempot

| المعيار               | التطبيق في Input Engine                                                             |
| --------------------- | ----------------------------------------------------------------------------------- | ----------- | ---------- |
| i18n-Only Rule        | لا نص صلب في المحرك — كل الأسئلة ورسائل الخطأ والتسميات من ملفات i18n               |
| Result Object Pattern | يُعيد دائماً `{ success: true, data }` أو `{ success: false, reason: 'timeout'      | 'cancelled' | 'error' }` |
| UX Helpers            | يستخدم ux-helpers/keyboards لبناء الأزرار وux-helpers/feedback لرسائل الخطأ الموحدة |
| Zod Validation        | كل Schema يُحقق عبر Zod — لا custom validation خارج الـ Schema                      |
| Session Integration   | يُخزن تقدم النموذج في `activeConversation` + `metadata` بـ SessionManager           |

### 7.3.6 قاعدة الإلغاء العالمي والـ Timeout

> ⚠️ كل نموذج يدعم الإلغاء والـ Timeout تلقائياً — لا يحتاج كل موديول لكتابة هذا المنطق.

| الآلية                          | السلوك                                                               |
| ------------------------------- | -------------------------------------------------------------------- |
| أمر `/cancel` في أي لحظة        | يُلغي النموذج فوراً ويُعيد `{ success: false, reason: 'cancelled' }` |
| formTimeout (افتراضي: 10 دقائق) | مدة انتظار قابلة للتعديل لكل نموذج                                   |
| fieldTimeout (اختياري)          | مدة انتظار لحقل واحد محدد — مفيد لحقول الموقع أو الصوت               |
| partialSave عند الإلغاء         | إذا فعّله الموديول: يُحفظ ما أدخله المستخدم في Redis ليُستأنف لاحقاً |

### 7.3.7 نمط الحفظ الجزئي — Partial Save

للنماذج الطويلة — يحفظ المحرك تقدم المستخدم في Redis تلقائياً بعد كل حقل.

| الإعداد في Schema              | السلوك                                            |
| ------------------------------ | ------------------------------------------------- |
| `partialSave: false` (افتراضي) | لا حفظ — النموذج يبدأ من الصفر دائماً             |
| `partialSave: true`            | يُحفظ تقدم الحقول في Redis — يُستأنف من آخر حقل   |
| `partialSaveTTL`               | مدة الاحتفاظ بالبيانات الجزئية (افتراضي: 24 ساعة) |

### 7.3.8 حقل الاستخراج الذكي — AIExtractorField

يسمح للمستخدم بإرسال رسالة صوتية أو نصية حرة (مثال: "اسمي أحمد وعمري 30 وأسكن في القاهرة") ويستخدم ai-core لاستخراج قيم عدة حقول دفعة واحدة.

| الحالة                        | السلوك                                                              |
| ----------------------------- | ------------------------------------------------------------------- |
| AI Provider متاح              | يستخرج القيم ويعرضها للمستخدم للتأكيد قبل قبولها — لا يقبل تلقائياً |
| AI Provider متعطل (راجع 17.5) | يتحول تلقائياً لإدخال يدوي حقلاً حقلاً — يُبلَّغ المستخدم بـ i18n   |
| استخراج جزئي (بعض الحقول فقط) | يقبل المستخرَج ويسأل عن الحقول الناقصة يدوياً                       |
| رفض المستخدم للاستخراج        | يتيح إعادة الإدخال يدوياً                                           |

### 7.3.9 نمط الاستخدام في الموديول

| الخطوة             | الكود / الإجراء                                                                                                                   |
| ------------------ | --------------------------------------------------------------------------------------------------------------------------------- |
| 1 — تفعيل الحزمة   | `features.hasInputEngine: true` في `module.config.ts`                                                                             |
| 2 — تعريف Schema   | `const mySchema = InputEngine.schema({ name: { type: 'ShortText', i18nKey: 'fields.name' }, age: { type: 'Integer', min: 18 } })` |
| 3 — الاستدعاء      | `const result = await InputEngine.runForm(ctx, mySchema, { timeout: 600 })`                                                       |
| 4 — معالجة النتيجة | `if (result.success) { await repo.save(result.data) } else { /* handle cancel/timeout */ }`                                       |

> ✅ لا يحتاج الموديول لمعرفة أي شيء عن grammY Conversations أو Zod internals — كل التعقيد محجوز داخل input-engine.

# 8. نظام التعدد اللغوي ومحرك CMS

> **In a Nutshell:** قاعدة ذهبية: لا نص مكتوب مباشرة في الكود. محرك CMS ديناميكي يتيح تعديل النصوص من لوحة التحكم فوراً عبر i18next backends + cache-manager بدلاً من resolver مخصص.

| الحزمة     | المسار                 | تعتمد على                       | اختياري                          |
| ---------- | ---------------------- | ------------------------------- | -------------------------------- |
| i18n-core  | `/packages/i18n-core`  | i18next                         | لا                               |
| cms-engine | `/packages/cms-engine` | i18next backends, cache-manager | نعم (`TEMPOT_DYNAMIC_CMS=false`) |

> 📐 **ADR-014:** اعتماد i18next backend plugins بدلاً من بناء CMS resolver مخصص. i18next موجود أصلاً في المشروع ويوفر plugins جاهزة لـ Redis/DB backend.
> 🔄 **تغيير عن v10:** كان يبني resolver مخصص بالكامل (cache + fallback + sync).

## 8.1 القاعدة الذهبية

> ⚠️ لا يُسمح بأي نص مكتوب مباشرة في الكود المصدري تحت أي ظرف. كل نص يجب أن يأتي من ملفات الترجمة عبر دالة الترجمة الموحدة. هذه القاعدة إلزامية وتُفحص تلقائياً عند التشغيل.

## 8.2 اللغات المدعومة

- العربية والإنجليزية مدعومتان افتراضياً.
- اللغة الافتراضية تُحدد عبر متغير البيئة `DEFAULT_LANGUAGE`.
- كل مستخدم يغير لغته من داخل البوت وتُحفظ في قاعدة البيانات.
- إضافة لغة جديدة تتطلب فقط إنشاء مجلد الترجمات دون أي تعديل في الكود.

## 8.3 هيكل ملفات الترجمة

| الحالة                                     | اللغة المستخدمة                    |
| ------------------------------------------ | ---------------------------------- |
| مستخدم حدد لغته                            | لغة المستخدم المختارة              |

## 8.5 محرك إدارة المحتوى والترجمات الديناميكي

### 8.5.1 التفعيل والسلوك عند الإيقاف

| الحالة                                 | سلوك النظام                                                            |
| -------------------------------------- | ---------------------------------------------------------------------- |
| `TEMPOT_DYNAMIC_CMS=true`              | المحرك نشط — النصوص المعدلة تتقدم على JSON الثابتة عبر i18next backend |

| `features.hasDynamicCMS=false` لموديول | لا تظهر مفاتيحه في لوحة التحكم حتى لو المحرك مفعل                      |

### 8.5.3 طبقة التخزين — عبر cache-manager

> 🔄 **تغيير عن v10:** بدلاً من بناء Hybrid Cache مخصص في cms-engine، يستخدم المحرك حزمة `cache-layer` المركزية (القسم 3.2).

| الطبقة                   | الدور                         | التفاصيل                                                       |
| ------------------------ | ----------------------------- | -------------------------------------------------------------- |
| ملفات `/locales/` (JSON) | Source of Truth البنيوي       | تحدد المفاتيح — لا تُعدَّل من لوحة التحكم أبداً                |
| PostgreSQL               | Source of Truth للقيم المعدلة | يُخزن: key, locale, value_new, value_old, updatedBy, updatedAt |
| cache-manager (Redis)    | Cache للأداء                  | استرجاع < 2ms — يُمسح فوراً عند التحديث                        |

> ⚠️ **قاعدة صارمة:** لوحة التحكم تُعدّل القيم فقط — لا تُضيف مفاتيح جديدة ولا تحذف موجودة. إضافة مفتاح أو حذفه عملية تطويرية تمر بـ SpecKit حصراً.

### 8.5.4 سلسلة التراجع — Resolution Fallback

| الأولوية   | المصدر                         | الشرط                                                |
| ---------- | ------------------------------ | ---------------------------------------------------- |
| 1 — الأعلى | cache-manager (Redis)          | `TEMPOT_DYNAMIC_CMS=true` والمفتاح معدَّل ديناميكياً |
| 2          | Static JSON (لغة المستخدم)     | ملف ar.json أو en.json                               |
| 3          | Static JSON (اللغة الافتراضية) | المفتاح موجود في DEFAULT_LANGUAGE فقط                |
| 4 — الأدنى | Error Key + WARN               | المفتاح غير موجود — يُعرض كنص ويُسجَّل WARN في Pino  |

### 8.5.5 دورة حياة تحديث النص — Event-Driven

| الخطوة                 | المنفذ                 | الوصف                                                                         |
| ---------------------- | ---------------------- | ----------------------------------------------------------------------------- |
| 1 — التعديل            | ADMIN في Dashboard     | يعدّل قيمة مفتاح موجود                                                        |
| 2 — الحفظ              | cms-engine             | يحفظ في PostgreSQL مع before/after/updatedBy ويُسجل في Audit Log (القسم 10.3) |
| 3 — إطلاق الحدث        | cms-engine → Event Bus | يُطلق `cms.translation.updated`                                               |
| 4 — Cache Invalidation | cms-engine listener    | يمسح المفتاح القديم من cache-manager ويُحمِّل الجديد                          |
| 5 — التطبيق الفوري     | كل مستخدم              | أي رسالة ترد في الثانية التالية تشاهد النص الجديد                             |

### 8.5.6 المفاتيح المحمية — Protected Keys

| نوع المفتاح المحمي  | أمثلة                                      |
| ------------------- | ------------------------------------------ |
| النصوص القانونية    | `privacy.policy.*`, `terms.of.service.*`   |
| تحذيرات الأمان      | `security.warning.*`, `auth.blocked.*`     |
| رسائل GDPR          | `gdpr.consent.*`, `data.deletion.*`        |
| رسائل النظام الحرجة | `system.maintenance.*`, `critical.error.*` |

SUPER_ADMIN يمكنه إضافة مفاتيح لقائمة الحماية — لكن لا يمكن لأحد تعديل المفاتيح المحمية مهما كان دوره.

### 8.5.7 دعم المتغيرات الديناميكية — Interpolation

| الصيغة                      | الوصف                                            |
| --------------------------- | ------------------------------------------------ |
| `{{variable}}`              | متغير بسيط — يُحقن من البيانات المُمررة لـ `t()` |
| `{{count}}` (Pluralization) | صيغة الجمع التلقائية عبر i18next                 |
| `{{variable, format}}`      | تنسيق متقدم: تاريخ، رقم، عملة                    |

> ⚠️ **حماية Interpolation:** المحرك يُعقم (Sanitize) قيم المتغيرات قبل الحقن عبر sanitize-html لمنع XSS.

### 8.5.8 اكتشاف المفاتيح عند التشغيل — Key Discovery

| الحالة                              | سلوك المحرك                                  |
| ----------------------------------- | -------------------------------------------- |
| مفتاح جديد في JSON غير موجود في DB  | يُسجَّل في DB تلقائياً ويُتاح للتعديل        |
| مفتاح محذوف من JSON لكن موجود في DB | يُصنَّف كـ orphaned key ويُنبَّه SUPER_ADMIN |
| مفتاح في DB بدون قيمة في JSON       | يُسجَّل WARN في Pino                         |

### 8.5.9 استعادة النص السابق — Rollback

| الإجراء                | السلوك                                |
| ---------------------- | ------------------------------------- |
| Rollback لتعديل واحد   | يستعيد قيمة before من آخر تعديل       |
| Rollback لتاريخ محدد   | يستعيد القيمة كما كانت في تاريخ بعينه |
| Rollback كامل للموديول | يُعيد كل المفاتيح لقيم JSON الأصلية   |

### 8.5.10 فحص اكتمال الترجمات

| الأمر                            | الوصف                               |
| -------------------------------- | ----------------------------------- |
| `pnpm cms:check`                 | يفحص اكتمال كل مفتاح في جميع اللغات |
| `pnpm cms:check --module {name}` | فحص موديول محدد                     |
| `pnpm cms:report`                | تقرير بالمفاتيح الناقصة             |
| `pnpm cms:sync`                  | مزامنة JSON مع DB                   |

> ✅ يُشغَّل `pnpm cms:check` إلزامياً في CI/CD عند كل PR يُعدِّل `/locales/`.

## 11.3 محرك السياق الإقليمي — Regional Context & Formatting Engine

> 📋 `/packages/regional-engine` حزمة Core مركزية بثلاثة مكونات: `timezone/` للتوقيت، `formatter/` للعملات والأرقام، `geo/` للبيانات الجغرافية. مصر (EG) هي الدولة الافتراضية.

> 📐 🆕 **ADR-024:** البيانات الجغرافية تُولَّد من `countries-states-cities-database` (7k+ ⭐، 153k+ مدينة، 19 لغة بما فيها العربية) بدلاً من بنائها يدوياً — البديل المرفوض: كتابة بيانات 27 محافظة مصرية يدوياً وصيانتها.

# 13. معايير تجربة المستخدم ومحرك البحث (UX Rules)

> **In a Nutshell:** معايير صارمة للأزرار والنصوص والرسائل. القاعدة الذهبية: تعديل الرسالة الموجودة دائماً بدلاً من إرسال جديدة. محرك بحث وتصفية متقدم يبني الفلاتر عبر Prisma مباشرة.

## 13.1 معايير الأزرار

## 13.2 قواعد النصوص العامة

- الرسائل الطويلة تُقسَّم لفقرات مع فراغات بينها.
- القوائم تستخدم رموزاً تعبيرية كنقاط بديلاً عن الشرطات.

- أزرار التأكيد والإلغاء دائماً في صف واحد.
- الرموز التعبيرية في بداية نص الزر لتسهيل القراءة.

## 13.3 دورة حياة الرسالة التفاعلية

> ⚠️ **القاعدة الذهبية:** تعديل الرسالة الموجودة دائماً بدلاً من إرسال رسالة جديدة.

المستخدم يضغط زر ← رسالة تحميل فورية (لا تتجاوز ثانية) ← تعديل نفس الرسالة بالنتيجة ← عرض الأزرار المناسبة للخطوة التالية.

## 13.4 رسائل الحالة الأربع

| الحالة | الرمز | النمط                  | مثال                               |

# 14. نظام الأحداث (Event Bus)

> **In a Nutshell:** ثلاثة مستويات (Local → Internal → External) مع ضمانات عدم الفقدان والترتيب. اتفاقية تسمية إلزامية: `module.entity.action`.

| الحزمة    | المسار                | تعتمد على                   | اختياري                        |
| --------- | --------------------- | --------------------------- | ------------------------------ |
| event-bus | `/packages/event-bus` | EventEmitter, Redis Pub/Sub | نعم (`TEMPOT_EVENT_BUS=false`) |

## 14.1 مستويات الأحداث

| المستوى  | الوصف                         | الآلية                           |
| -------- | ----------------------------- | -------------------------------- |
| Local    | أحداث داخل نفس الموديول       | Node.js EventEmitter — صفر تأخير |
| Internal | أحداث بين الموديولات المختلفة | Local Event Bus مركزي            |
| External | أحداث تحتاج scaling أو جدولة  | Redis Pub/Sub — at-most-once     |

## 14.2 اتفاقية تسمية الأحداث — إلزامية

> ⚠️ صيغة التسمية: `{module}.{entity}.{action}` — ثلاثة أجزاء دائماً، حروف صغيرة، نقطة كفاصل، الفعل في الآخر.

أمثلة صحيحة:

- `invoices.payment.completed`
- `users.user.role_changed`
- `system.backup.failed`
- `cms.translation.updated`
- `document.export.requested`

## 14.3 عقد الحدث الموحد

| الحقل     | الوصف                                    |
| --------- | ---------------------------------------- |
| eventId   | معرف فريد لكل حدث                        |
| eventName | اسم الحدث باتفاقية التسمية               |
| module    | الموديول المصدر                          |
| userId    | من أطلق الحدث — `null` للأحداث التلقائية |
| payload   | البيانات الخاصة بالحدث                   |
| timestamp | وقت إطلاق الحدث                          |
| level     | `local` أو `internal` أو `external`      |

## 14.4 ضمانات النظام

| الضمان         | التفصيل                                                                             |
| -------------- | ----------------------------------------------------------------------------------- |
| تسليم الأحداث  | Local/Internal: at-most-once (fire-and-forget). لا يوجد persistence أو retry تلقائي |
| ترتيب الأحداث  | يُعالَج بترتيب التسجيل ضمن نفس المُصدِر (synchronous handler execution)             |
| معالجة الفشل   | Handler فشل يُلتقط عبر try/catch ويُسجَّل كخطأ — لا يؤثر على باقي الـ handlers      |
| مستوى External | Redis Pub/Sub: at-most-once delivery — لا persistence بعد استلام Redis              |

## 15.6 إدارة التغييرات الكسرية — Breaking Schema Changes

> ⚠️ التعديلات الجوهرية على قاعدة البيانات تؤثر على المحادثات النشطة في Redis.

**استراتيجية Session Schema Versioning:**

كل بيانات محادثة تحمل حقل `_version: number`. عند تعديل Schema:

| الخطوة                      | الإجراء                                                  |
| --------------------------- | -------------------------------------------------------- |
| 1 — رفع رقم الإصدار         | زيادة `SESSION_SCHEMA_VERSION` في `module.config.ts`     |
| 2 — كتابة Migration Handler | دالة `migrateSession(oldData, fromVersion)`              |
| 3 — تطبيق عند القراءة       | SessionManager يقرأ `_version` ويُطبِّق handler تلقائياً |
| 4 — فترة التعايش            | الكود يدعم إصدارَين متتاليَين لمدة 24 ساعة               |
| 5 — تنظيف Redis             | بعد 24 ساعة: `pnpm session:cleanup`                      |

**قواعد إلزامية:**

- كل تغيير كسري يُوثَّق في ADR قبل التنفيذ.
- وضع الصيانة يُفعَّل قبل تطبيق أي Migration كسري في الإنتاج.
- النسخ الاحتياطي الكامل إلزامي قبل أي Migration كسري.

## 15.7 الاكتشاف التلقائي والتحقق

عند التشغيل يكتشف Module Registry كل الموديولات تلقائياً ويُشغّل Module Validator. الموديولات الأساسية (`isCore: true`) توقف التطبيق إن كانت غير مكتملة، والعادية تُتخطى مع تسجيل تحذير.

```yaml
# .github/workflows/claude-review.yml
name: Claude Code Review
on:
  pull_request:
    types: [opened, synchronize]
  issue_comment:
    types: [created]

jobs:
  review:
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write
      contents: read
    steps:
      - uses: actions/checkout@v4
      - uses: anthropics/claude-code-action@v1
        with:
          # خيار 1: API key (إذا لديك)
          # anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          # خيار 2: اشتراك Pro/Max عبر OAuth token (الموصى به)
          claude_code_oauth_token: ${{ secrets.CLAUDE_CODE_OAUTH_TOKEN }}

  security:
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write
      contents: read
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 2
      - uses: anthropics/claude-code-security-review@main
        with:
          comment-pr: true
          # خيار 1: API key
          # claude-api-key: ${{ secrets.ANTHROPIC_API_KEY }}
          # خيار 2: OAuth token من اشتراك Pro/Max
          claude-api-key: ${{ secrets.CLAUDE_CODE_OAUTH_TOKEN }}
```

# 24. نظام الاختبارات

> **In a Nutshell:** هرم اختبارات ثلاثي (Unit 70% + Integration 20% + E2E 10%) مع Vitest. معايير تغطية إلزامية تمنع البناء إن نقصت. كل خدمة خارجية تُحاكى في الاختبارات.

## 24.1 هرم الاختبارات

| المستوى           | الأداة                  | النسبة | ما يختبره                               |
| ----------------- | ----------------------- | ------ | --------------------------------------- |
| Unit Tests        | Vitest                  | 70%    | كل دالة في Services وHelpers وUtilities |
| Integration Tests | Vitest + Testcontainers | 20%    | Service + DB + Redis معاً               |
| E2E Tests         | Vitest + grammY Test    | 10%    | تدفقات المستخدم الحرجة كاملاً           |

## 24.2 ما هو إلزامي في كل موديول

| الملف                          | المستوى     | إلزامي                    |
| ------------------------------ | ----------- | ------------------------- |
| `{feature}.service.test.ts`    | Unit        | نعم — لكل وظيفة           |
| `{feature}.handler.test.ts`    | Unit        | نعم — لكل handler         |
| `{module}.integration.test.ts` | Integration | نعم — لكل موديول          |
| `{feature}.e2e.test.ts`        | E2E         | نعم — للتدفقات الحرجة فقط |

> 📄 **لاستراتيجية الاختبار الموسعة:** راجع ملف `docs/guides/TESTING-STRATEGY-EXTENDED.md` — يشمل Performance Testing, Chaos Testing, Contract Testing, وCI/CD Integration.

| المستوى       | الحد الأدنى | يمنع البناء إن نقص |
| ------------- | ----------- | ------------------ |
| Services      | 80%         | نعم                |
| Handlers      | 70%         | نعم                |
| Repositories  | 60%         | تحذير فقط          |

## 24.4 قواعد كتابة الاختبارات

| القاعدة     | التفصيل                                                                             |
| ----------- | ----------------------------------------------------------------------------------- |

## 25.3 Graceful Shutdown — إيقاف منظم

> 🆕 **قسم جديد — كان غائباً في v10.**

عند استقبال إشارة `SIGTERM` أو `SIGINT`:

| الخطوة | الإجراء                                       | المهلة   |
| ------ | --------------------------------------------- | -------- |
| 1      | إيقاف استقبال طلبات جديدة (Hono server close) | فوري     |
| 2      | إكمال الطلبات الجارية في grammY               | 10 ثوانٍ |
| 3      | إنهاء BullMQ workers عبر queue factory        | 15 ثانية |
| 4      | إغلاق اتصال Redis (ioredis quit)              | 5 ثوانٍ  |
| 5      | إغلاق اتصال Prisma ($disconnect)              | 5 ثوانٍ  |
| 6      | إغلاق اتصال Drizzle (pgvector pool)           | 5 ثوانٍ  |
| 7      | تسجيل `system.shutdown.completed` في Pino     | فوري     |

```typescript
process.on('SIGTERM', async () => {
  logger.info('Graceful shutdown initiated...');
  await honoServer.close();
  await bot.stop();
  await queueManager.closeAll();
  await redis.quit();
  await prisma.$disconnect();
  await drizzlePool.end();
  logger.info('Shutdown completed.');
  process.exit(0);
});
```

> ⚠️ إن تجاوزت المهلة الإجمالية 30 ثانية → `process.exit(1)` مع `FATAL` في Pino.

# 28. نظام الأمان المتقدم

> **In a Nutshell:** Security by Default + Redis Degradation Strategy لمعالجة نقطة الفشل الواحدة.

## 28.1 مبدأ Security by Default

كل طلب يمر تلقائياً بسلسلة أمان كاملة: `sanitize-html` → `ratelimiter` → `CASL` → `Zod` → `Business Logic` → `Audit Log`.

## 28.2 Redis Degradation Strategy

> 🆕 **قسم جديد — كان غائباً في v10. Redis نقطة فشل واحدة محتملة.**

| سيناريو الفشل                  | الاستراتيجية                                                             |
| ------------------------------ | ------------------------------------------------------------------------ |
| Redis معطل مؤقتاً (< 30 ثانية) | ioredis auto-reconnect كل 2 ثانية                                        |
| Redis معطل ممتداً (> 30 ثانية) | الجلسات → in-memory مؤقت، cache-manager → DB مباشرة، BullMQ → ينتظر      |
| Redis عاد بعد انقطاع           | ioredis يعيد الاتصال، cache-manager تُعيد البناء تدريجياً، BullMQ يستأنف |

> ⚠️ **قواعد إلزامية:**
>
> - كل عملية Redis تُلف بـ try/catch مع fallback.
> - SessionManager يحتفظ بنسخة في DB — يقرأ منها عند سقوط Redis.
> - تنبيه فوري لـ SUPER_ADMIN عند كل فشل.
> - cache-manager wrapper يدعم `mode: 'redis-only' | 'db-fallback' | 'memory-fallback'`.
