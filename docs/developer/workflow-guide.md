## المرحلة الأولى — المواصفات والتوثيق (SpecKit)

### الخطوة 1: تأسيس مبادئ المشروع (مرة واحدة)

```
/speckit.constitution
```

**المخرج:** `.specify/memory/constitution.md`

يحتوي على: معايير الكود، CASL RBAC، i18n-Only، Repository Pattern، Event Bus، ADR-030 limits.

> ⚠️ تُنفَّذ مرة واحدة فقط عند بدء المشروع. لا تُعاد.

---

### الخطوة 2: كتابة المواصفات

```
/speckit.specify أريد بناء موديول إدارة المستخدمين مع 4 أدوار وإدارة حالات الحسابات
```

**المخرج:** `.specify/specs/001-user-management/spec.md`

> 💡 ركّز على "ماذا ولماذا" — لا تذكر تقنيات أو كود.

---

### الخطوة 3: التوضيح والحالات الحدية

```
/speckit.clarify
```

**المخرج:** قسم Clarifications يُضاف إلى `spec.md`

> ❌ **ممنوع التخطي** — هذه الخطوة تكشف الحالات الحدية التي تسبب أخطاء لاحقاً.

---

### الخطوة 4: إنشاء الخطة التقنية

```
/speckit.plan سنستخدم Prisma 7 مع PostgreSQL، CASL للصلاحيات، neverthrow لمعالجة الأخطاء
```

**المخرج:**
- `plan.md` — خطة التنفيذ
- `data-model.md` — نماذج البيانات
- `research.md` — بحث تقني عن المكتبات

### الخطوة 5: التحقق من تناسق الخطة

```
/speckit.validate
```

**المخرج:** تقرير تحقق — يكشف التعارضات مع الموديولات الموجودة.

> ⚠️ حل كل التعارضات قبل الانتقال للمرحلة الثانية.

## المرحلة الثانية — التنفيذ والجودة (superpowers)

### الخطوة 6: التصميم المعماري

في Claude Code أو Gemini CLI:
```
/brainstorming

أريد تصميم موديول إدارة المستخدمين حسب المواصفات في .specify/specs/001-user-management/spec.md
```

**ماذا يحدث:**
1. أسئلة سقراطية لتنقية الفكرة
2. استكشاف البدائل المعمارية
3. عرض التصميم بأقسام قصيرة للموافقة
4. حفظ وثيقة التصميم

**المخرج:** `docs/superpowers/specs/{date}-{feature}.md`

---

### الخطوة 7: إنشاء بيئة عمل معزولة

```
/using-git-worktrees
```

**ماذا يحدث:**
1. إنشاء branch جديد: `feat/user-management`
2. إنشاء worktree معزول
3. تشغيل setup المشروع
4. التحقق من أن كل الاختبارات الحالية تمر

---

### الخطوة 8: كتابة خطة التنفيذ

```
/writing-plans
```

**ماذا يحدث:**
- تقسيم العمل لمهام 2-5 دقائق
- كل مهمة تحتوي: مسارات ملفات دقيقة، كود كامل، خطوات تحقق
- ترتيب حسب التبعيات

**المخرج:** `docs/superpowers/plans/{date}-{feature}.md`

---

### الخطوة 9: التنفيذ مع TDD

```
/executing-plans docs/superpowers/plans/{date}-{feature}.md
```

**ماذا يحدث:**
- تنفيذ مهمة تلو أخرى مع نقاط تحقق
- **TDD إلزامي في كل مهمة:**
  1. 🔴 RED: كتابة اختبار يفشل
  2. 🟢 GREEN: كتابة أقل كود لنجاحه
  3. 🔵 REFACTOR: تحسين بدون تغيير السلوك
  4. ✅ COMMIT

> ⚠️ superpowers يحذف أي كود كُتب بدون اختبار أولاً.

---

### الخطوة 10: مراجعة الكود

```
/requesting-code-review
```

**ماذا يحدث:**
- مراجعة ضد الخطة الأصلية
- تصنيف المشاكل: Critical (يمنع الإكمال) / Warning / Info
- حل المشاكل الحرجة إلزامي قبل الدمج

---

### الخطوة 11: إغلاق الفرع

```
/finishing-a-development-branch
```

**الخيارات:**
1. **Merge locally** — دمج في main محلياً
2. **Create PR** — رفع لـ GitHub وإنشاء Pull Request
3. **Keep working** — مواصلة العمل
4. **Discard** — حذف كل التغييرات

## المسار المختصر — Hotfix Track

> للأخطاء الحرجة في الإنتاج فقط (P0/P1):

| # | الإجراء |
|---|---|
| 1 | توثيق الخطأ في Issue مع وسم `hotfix` |
| 2 | إصلاح مباشر مع اختبار وحدوي إلزامي |
| 3 | PR ببادئة `security:` أو `fix!:` |
| 4 | بعد النشر: `spec.md` بأثر رجعي خلال 48 ساعة |

**الشروط:** موافقة SUPER_ADMIN + الإصلاح ≤ 50 سطر.
