# 10 - خطة الإصلاح المرحلية

> كل مرحلة تُنفَّذ ضمن منهجية SpecKit + Superpowers الحالية، باحترام كامل لقواعد الدستور، إطار الأدوار، وبوابات الجودة. لا مسار تنفيذ موازٍ.

## النظرة العامة

| المرحلة                 | الإطار الزمني المقترح | المعرّفات                            | الهدف الرئيسي                                                |
| ----------------------- | --------------------- | ------------------------------------ | ------------------------------------------------------------ |
| 1. عاجلة (P0)            | 0–2 أسبوع             | I004، I011، I012                     | إغلاق Spec #057 + تأمين الأسرار + تفعيل gitleaks.            |
| 2. قصيرة المدى (P1)      | 2–6 أسابيع            | I001، I002، I003، I006، I007، I013    | إغلاق ديون الجودة وSpec #058 وتعزيز coverage الحرج.          |
| 3. متوسطة المدى (P2)     | 6–12 أسبوع            | I005، I008، I010، I015               | توحيد الوثائق، تنظيف القوالب، تحديث الدستور.                |
| 4. طويلة المدى (P3)      | 12+ أسبوع             | I009، I014، تحسينات DX                | تقليل drift عبر سكربتات context + توثيق specs ملغاة.        |

## المرحلة 1: العاجلة (P0)

### الهدف

إغلاق فجوات النشر الإنتاجية وحماية الأسرار التشغيلية بأقل تغيير ممكن للكود الإنتاجي.

### المهام (بترتيب التنفيذ)

#### المهمة 1.1 — إكمال Spec #057 (Production Delivery Hardening)

| البند              | التفاصيل                                                                                                                            |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------- |
| المعرّف            | I012                                                                                                                                |
| السبب              | لا يمكن إصدار `v1.0` بدون أدلة staging حقيقي + backup/restore + go/no-go.                                                            |
| الأثر              | يفتح إمكانية النشر الإنتاجي الموقوف.                                                                                                |
| مكان التنفيذ       | `specs/057-production-delivery-hardening/` Phases 5–7 + T032.                                                                       |
| المخاطر             | اعتماد على بيئة staging حقيقية وبيانات هدف.                                                                                          |
| طريقة التنفيذ      | متابعة Spec #057 الحالي عبر Superpowers (لا spec جديد). كل phase ينتج evidence أصلية تحت `docs/operations/`.                          |
| معايير القبول      | (a) staging smoke test ناجح وموثَّق؛ (b) backup/restore evidence مع dataset مقصود؛ (c) rollback أو forward-fix evidence؛ (d) go/no-go log رسمي. |
| طريقة التحقق       | `pnpm spec:validate`؛ ROADMAP يظهر Spec #057 = Complete؛ commit/PR مرفق بأدلة.                                                       |

#### المهمة 1.2 — تفعيل secrets scanning في CI (gitleaks)

| البند              | التفاصيل                                                                                                                            |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------- |
| المعرّف            | I011                                                                                                                                |
| السبب              | حماية defense-in-depth ضد تسرب أسرار في commits مستقبلية.                                                                            |
| الأثر              | يقطع طريق تسرب `.env` أو `BOT_TOKEN` بشريًا.                                                                                          |
| مكان التنفيذ       | `.github/workflows/ci.yml` (job جديد `secrets-scan`).                                                                                |
| المخاطر             | false positives قد تحجب PRs مؤقتًا.                                                                                                  |
| طريقة التنفيذ      | spec جديد مقترح `059-security-hardening-followups` بمسار: specify → clarify → plan → analyze → tasks → executing-plans (TDD على الـ workflow عبر PR PoC). |
| معايير القبول      | gitleaks step يفشل على commit صناعي يحوي `BOT_TOKEN=xxxx`، ويمر على main current بدون مشاكل.                                          |
| طريقة التحقق       | اختبار PR قصير يحوي token وهمي ⇒ يفشل CI كما هو متوقع، يُغلق بدون merge.                                                              |

#### المهمة 1.3 — توثيق دورة تدوير الأسرار

| البند              | التفاصيل                                                                                                            |
| ------------------ | ------------------------------------------------------------------------------------------------------------------- |
| المعرّف            | I004                                                                                                                |
| السبب              | تحويل Rule XXXI من مبدأ إلى عملية يومية.                                                                            |
| الأثر              | يقلل blast radius عند تسرب محتمل.                                                                                  |
| مكان التنفيذ       | `docs/security/secrets-rotation.md` (مستند جديد) + إشارة في `docs/security/SECURITY.md`.                              |
| المخاطر             | لا شيء (وثيقة).                                                                                                     |
| طريقة التنفيذ      | spec docs-only ضمن `059-security-hardening-followups`؛ يلتزم بـ Rule L (Code-Docs Parity).                            |
| معايير القبول      | runbook يحدد: BOT_TOKEN كل 90 يومًا، PROTECTED_DATA_* كل 180 يومًا، WEBHOOK_SECRET_TOKEN عند الحاجة. مع نقطة فحص في `pnpm tempot doctor`. |
| طريقة التحقق       | مراجعة محتوى الـ runbook + `pnpm tempot doctor` يطبع تذكيرًا.                                                          |

### علاقة المرحلة بمنهجية العمل

- المهمة 1.1 تستمر داخل Spec #057 الحالي → لا حاجة spec جديد.
- المهمتان 1.2 و1.3 تدخلان معًا في spec أمني واحد مقترح `059-security-hardening-followups` (يبدأ بعد إغلاق #058).
- جميع التغييرات تمر عبر TDD حيث ينطبق (workflow CI قابل للاختبار)، وReview Gate، وReconciliation Gate.

## خلاصة

- الخطة **تنفيذية بالكامل ضمن المنهجية الحالية**.
- لا تتطلب إصلاحات مدمرة أو إعادة هيكلة.
- معظم المهام صغيرة الحجم؛ الإصلاحات الكبيرة (#057 و#058) كانت بدأت قبل هذا التحليل.
- التركيز الأكبر على إثبات النشر وإغلاق ديون الجودة الصغيرة لتمكين إصدار v1.0 مدعومًا بأدلة.
