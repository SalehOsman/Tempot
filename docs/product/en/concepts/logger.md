## What is the Logger Package?

The `@tempot/logger` package provides two complementary logging systems: a technical logger for operational diagnostics and an audit logger for compliance-grade state change tracking. Both produce structured JSON and integrate with Tempot's session context for automatic user attribution.

## Two Logging Systems

| System    | Class                    | Purpose                                             | Output                               |
| --------- | ------------------------ | --------------------------------------------------- | ------------------------------------ |
| Technical | `logger` (Pino instance) | Operational diagnostics, debugging, error reporting | Structured JSON to stdout            |
| Audit     | `AuditLogger`            | State change tracking, compliance, traceability     | Persisted to AuditLog database table |

These serve different audiences. The technical logger helps developers diagnose issues. The audit logger provides a tamper-proof record of who changed what and when, required for enterprise compliance.

## Structured JSON Format

All technical logs are JSON objects produced by Pino. A typical log entry contains:

```json
{
  "level": 30,
  "time": 1711900000000,
  "pid": 12345,
  "hostname": "server-01",
  "userId": "user_abc123",
  "msg": "Order processed successfully"
}
```

The `userId` field is injected automatically via Pino's `mixin` option, which reads from `sessionContext` on every log call. No manual user attribution is needed.

## Log Levels

Pino provides six hierarchical log levels:

| Level   | Value | Usage                                          |
| ------- | ----- | ---------------------------------------------- |
| `trace` | 10    | Granular debugging (request/response payloads) |
| `debug` | 20    | Development-time diagnostic info               |
| `info`  | 30    | Normal operational events (default level)      |
| `warn`  | 40    | Recoverable issues, degraded operations        |
| `error` | 50    | Operation failures, caught errors              |
| `fatal` | 60    | Unrecoverable errors, process about to exit    |

The active level is controlled by the `LOG_LEVEL` environment variable, defaulting to `info`.

## Context Injection

The `sessionContext` from `@tempot/shared` is an `AsyncLocalStorage` instance carrying user session data through the async call stack. The logger reads it in two places:

- **Technical logger**: Pino's `mixin` function injects `userId` into every log entry
- **Audit logger**: `AuditLogger.log()` reads `userId` and `userRole` for attribution

This means any code running within a session context automatically produces attributed logs without passing user data through function parameters.

## PII Redaction

Sensitive data is redacted at two layers:

### Pino Built-in Redaction

Pino's native `redact` option strips values for keys matching `SENSITIVE_KEYS` (`password`, `token`, `secret`, `apiKey`, `creditCard`) from top-level log properties.

### Error Serializer Redaction

The `appErrorSerializer` recursively walks `err.details` and replaces any value whose key matches `SENSITIVE_KEYS` with `[REDACTED]`. This handles arbitrarily nested objects and arrays.

## No Double Logging

When an `AppError` is first serialized by `appErrorSerializer`, the serializer stamps `err.loggedAt = new Date()`. On subsequent encounters, the serializer recognizes the stamp and returns a minimal `{ message: 'Already logged', code, __redundant: true }` stub. This prevents the same error from producing duplicate log entries as it propagates up the call stack.

## Error Reference Codes

Non-success audit entries automatically receive a unique reference code in the format `ERR-YYYYMMDD-XXXX` (generated by `@tempot/shared`). This code links three things together: the user-facing error message, the audit log entry, and the Sentry event. The `appErrorSerializer` also surfaces `referenceCode` in technical logs when present on the error.

## Audit Logger

The `AuditLogger` persists state changes to the database via `AuditLogRepository`. Each entry records:

- **action**: The operation (e.g., `invoices.invoice.create`)
- **module**: The originating module
- **before/after**: JSON snapshots of the entity state
- **userId/userRole**: From session context or explicit override
- **status**: Defaults to `SUCCESS`
- **referenceCode**: Auto-generated for non-success statuses

Audit records are immutable. The `AuditLogRepository` rejects all update and delete attempts with an error result.
