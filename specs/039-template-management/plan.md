# Implementation Plan: Template Management Module

**Branch**: `039-template-management` | **Date**: 2026-05-10 | **Spec**: `specs/039-template-management/spec.md`

## Summary

Implement a full-featured template management module that combines CRUD,
lifecycle workflow, versioning, search, import/export, ratings, notifications,
and CMS integration. This is the first Product-type module and the most
package-integrated module in the Tempot catalog.

## Technical Context

**Language/Version**: TypeScript 5.9.3 strict mode
**Primary Dependencies**: grammY 1.41.x, Prisma 7.x, neverthrow 8.2.0, Zod
**Storage**: PostgreSQL 16 (6 new tables + 1 junction table)
**Testing**: Vitest 4.1.0 + Testcontainers
**Target Platform**: Node.js 22.12+ / Telegram Bot API
**Project Type**: Business module in monorepo
**Performance Goals**: < 500ms CRUD, < 1s search
**Constraints**: RBAC on every operation, bilingual (ar/en), soft-delete only
**Scale/Scope**: ~1000 templates catalog

## Architecture Impact

- **New database tables**: 6 tables + 1 junction (see data-model.md)
- **New Prisma migration**: Required for all new tables
- **New package dependencies**: None вЂ” uses existing @tempot/* packages
- **No changes to existing packages**: All integration via public contracts
- **Event bus**: 6 new event types published
- **Notifier integration**: 3 notification triggers

## Implementation Strategy

### Layer 1: Foundation (Types, Contracts, Config)

1. Module scaffolding: `package.json`, `tsconfig.json`, `vitest.config.ts`,
   `module.config.ts`, `module.manifest.ts`, `README.md`, `index.ts`
2. Type definitions: all entities, enums, and action types
3. Zod schemas: template content, bundle, create/update validation
4. Lifecycle transitions: state machine definition + guards
5. Event contracts: names and payload types
6. Abilities: RBAC rules for GUEST, USER, ADMIN, SUPER_ADMIN
7. Locale files: `ar.json` and `en.json` with all UI text

### Layer 2: Persistence (Repositories + Migration)

8. Prisma schema for all 7 tables
9. Template repository: CRUD, status update, search query builder
10. Version repository: create snapshot, list by template
11. Category repository: CRUD with hierarchy, reorder
12. Tag repository: CRUD, usage count, autocomplete
13. Rating repository: upsert, average recalculation
14. Subscription repository: create, delete, list by template/user

### Layer 3: Business Logic (Services)

15. Template service: CRUD orchestration, slug generation, validation
16. Lifecycle service: guarded transitions, completeness check
17. Version service: snapshot creation, version bump logic, diff suggestion
18. Clone service: deep copy, subscription auto-create, usage increment
19. Search service: search-engine adapter, filter builder, sort, pagination
20. Category service: hierarchy validation, depth guard, reorder
21. Tag service: create-or-find, slug generation, autocomplete
22. Rating service: upsert, average recalculation, sort support
23. Subscription service: follow/unfollow, list subscribers
24. Export service: JSON bundle builder, PDF document request
25. Import service: bundle validation, category/tag resolution, DRAFT creation

### Layer 4: Presentation (Commands, Menus, Handlers)

26. Menu factories: all 7 menu factories with Inline Keyboard builders
27. Commands: `/templates`, `/new_template`, `/import_template`
28. Callback handler: route all `tmpl:*` callbacks to services
29. Text handler: wizard inputs, search queries, rejection reasons
30. Notification handler: listen for template events, enqueue via notifier

### Layer 5: Testing

31. Unit tests: all services, schemas, lifecycle transitions, menu factories
32. Integration tests: CRUD flow, lifecycle flow, clone flow, search flow
33. Boundary and checklist: `pnpm boundary:audit` + `pnpm module:checklist`

### Layer 6: Documentation and Gates

34. README with module overview, commands, architecture, and testing
35. Run all quality gates: lint, build, test:unit, spec:validate,
    boundary:audit, module:checklist, cms:check

## Key Design Decisions

| Decision | Rationale |
| --- | --- |
| JSONB for template content | Self-contained bundle, schema evolution without migration, atomic version snapshots |
| Pure function state machine | 4 states, no external library needed |
| Separate TemplateVersion table | Immutable snapshots, independent of draft edits |
| PostgreSQL tsvector for search | Native full-text with relevance ranking, no external search service |
| Auto-subscribe on clone | Natural subscription вЂ” users who clone care about updates |
| Slug generation | URL-safe identifiers for future dashboard/API use |
| Soft-delete with is_deleted | Audit trail preservation, consistent with Tempot convention |

## Risk Assessment

| Risk | Mitigation |
| --- | --- |
| JSONB schema drift | Zod validation on every read/write |
| Category depth bugs | Depth column + CHECK constraint + service guard |
| Concurrent lifecycle transitions | Optimistic locking via updatedAt |
| Large import bundles | 5MB file size limit + streaming validation |
| Search performance at scale | tsvector index + pagination + cache |
