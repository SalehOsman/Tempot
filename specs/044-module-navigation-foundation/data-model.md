# Data Model: Module Navigation Foundation

## NavigationContribution

Represents one module-owned entry that can appear in a navigation surface.

Fields:

- `id`: Stable unique contribution identifier.
- `ownerModule`: Module that owns the contribution.
- `surface`: Navigation surface where the contribution can appear.
- `labelKey`: i18n key for the visible label.
- `callbackAction`: Callback action or namespace that the owner handles.
- `requiredRoles`: Roles allowed to see the contribution.
- `requiredCapabilities`: Capability flags that must be available.
- `order`: Stable ordering value for predictable menu layout.
- `enabledByDefault`: Whether the contribution is active when the module is enabled.

Validation rules:

- `id` must be unique across all active contributions.
- `callbackAction` must have one active owner.
- `labelKey` must exist in all required locales.
- `ownerModule` must match an enabled and validated module.
- Role requirements must not expose admin-only actions to standard users.

## CallbackOwnership

Represents the owner responsible for handling an action generated by inline navigation.

Fields:

- `callbackAction`: Stable action or namespace.
- `ownerModule`: Module responsible for handling it.
- `handlerScope`: Whether ownership applies to an exact action or namespace.
- `fallbackKey`: i18n key used when stale callbacks can no longer be handled.

Validation rules:

- No two enabled modules may own the same exact action.
- Namespace ownership must not overlap another exact action unless explicitly allowed by validation.
- Ownership must be registered before polling or webhook serving begins.

## NavigationSurface

Represents a rendered menu or help view for a specific user and context.

Fields:

- `surfaceId`: Stable surface name such as main menu or help.
- `userRole`: Role used for visibility filtering.
- `locale`: Locale used to resolve labels.
- `contributions`: Ordered contributions visible to the user.
- `generatedAt`: Time the surface was assembled.

Validation rules:

- A surface may render with zero optional contributions but must still provide a valid user response.
- Contributions must be filtered by role and availability before rendering.

## CapabilityAvailability

Represents whether a contribution can be shown and handled.

Fields:

- `moduleName`: Module that owns the capability.
- `enabled`: Whether module enablement allows it.
- `validated`: Whether startup validation passed.
- `runtimeReady`: Whether required runtime dependencies are available.
- `reasonKey`: Optional i18n key explaining unavailability.

Validation rules:

- Unavailable contributions are not shown in fresh menus.
- Stale callbacks for unavailable contributions receive a localized unavailable-action response.

## State Transitions

Navigation contribution:

```text
declared -> validated -> visible -> handled
declared -> validation_failed -> hidden
visible -> owner_disabled -> stale_unavailable
visible -> role_changed -> stale_unavailable
```

Callback ownership:

```text
declared -> registered -> routable
declared -> duplicate_detected -> startup_rejected
registered -> owner_unavailable -> stale_unavailable
```
